# Dashboard Validation Principles

이 문서는 `src/dashboard_app.py`와 관련 대시보드 변경을 검토할 때 사용하는 하네스 검증 원칙이다.
목적은 "무엇을 고쳐야 하는가"를 취향이 아니라 재현 가능한 기준으로 판단하는 것이다.

## 1. North Star

대시보드는 예쁜 화면이 아니라 제품 의사결정을 돕는 분석 인터페이스여야 한다.
따라서 모든 수정은 아래 질문 중 하나에 더 잘 답해야 한다.

- 지금 어떤 제품/브랜드/국가에서 문제가 커지고 있는가?
- 그 문제는 실제 고객 반응인가, 표본 왜곡인가?
- 이 화면을 본 사람이 다음 액션을 결정할 수 있는가?

위 질문에 기여하지 않는 시각적 장식, 과도한 문구, 중복 정보는 우선 제거 대상이다.

## 2. Dashboard Constitution

### 2.1 Truth First
- 화면 숫자는 원본 집계 로직과 일치해야 한다.
- KPI, 차트, 표, 다운로드 파일이 서로 다른 숫자를 보여주면 실패다.
- 같은 필터 조건에서 카드 수치와 상세 표본의 방향성이 어긋나면 실패다.

### 2.2 Filter Consistency
- 국가, 브랜드, 제품, 감성, CEJ, 키워드 필터는 모든 관련 위젯에 동일하게 적용되어야 한다.
- 한 필터를 바꿨을 때 일부 패널만 갱신되거나 이전 상태를 유지하면 실패다.
- 빈 결과일 때는 조용히 틀린 데이터를 보여주지 말고 명시적으로 빈 상태를 보여줘야 한다.

### 2.3 Decision Usefulness
- 각 섹션은 "그래서 무엇을 해야 하는가"를 읽을 수 있어야 한다.
- 대표 댓글, 키워드, CEJ 표는 서로 보완 관계여야 하며 같은 의미를 다른 표현으로 반복하면 안 된다.
- 새 위젯을 추가할 때는 기존 위젯 하나를 대체하거나 명확히 다른 판단 질문을 담당해야 한다.

### 2.4 Explainability
- 계산식이 있는 지표는 최소한 문구 또는 구조상 해석이 가능해야 한다.
- 고객 신뢰도, 비율, 분리 비율처럼 해석이 필요한 값은 설명 없이 놓이면 실패다.
- AI 재분석 결과가 표시되면 원문, 번역, 맥락, 분류 이유 중 최소한 일부가 함께 보여야 한다.

### 2.5 Performance Is Product Quality
- 대시보드가 느리거나 자주 꺼지면 분석 품질과 동일하게 실패로 본다.
- 첫 로드, 필터 변경, 다운로드 동작은 사용자가 기다릴 만한 수준이어야 한다.
- 대표 댓글/번역/워드클라우드처럼 비싼 작업은 범위를 제한하고 캐시를 사용해야 한다.

### 2.6 Safe Handling of User Text
- 원문 댓글은 분석 목적 범위 내에서만 보여준다.
- 불필요한 개인식별 정보, 작성자 식별 정보, 과도한 원문 노출은 피한다.
- 로그나 예외 메시지에 원문 전체를 그대로 남기면 실패다.

## 3. Validation Lenses

대시보드 수정은 아래 6개 렌즈로 반드시 검토한다.

### 3.1 Data Lens
검증 질문:
- 카드, 차트, 표, 다운로드의 숫자가 같은 필터 기준으로 계산되는가?
- `valid`, `excluded`, `neutral` 처리 기준이 모든 패널에서 일관적인가?
- KR/US, 브랜드, 제품명이 로컬라이즈된 표시와 내부 코드 사이에서 꼬이지 않는가?

증거:
- 집계 함수 확인
- 작은 샘플 DataFrame 기반 테스트
- 다운로드 엑셀과 화면 수치 대조

### 3.2 UX Lens
검증 질문:
- 사용자가 10초 안에 현재 상태를 이해할 수 있는가?
- 필터 순서와 기본값이 실제 사용 시나리오에 맞는가?
- 빈 상태, 데이터 없음, 로딩 중 상태가 명확한가?

증거:
- Lite/Full 모드 실제 확인
- 초기 진입 화면 스크린 점검
- 빈 데이터 조건 강제 테스트

### 3.3 Insight Lens
검증 질문:
- 이 화면이 행동 결정을 돕는가?
- 정보가 많아 보이지만 실제로는 같은 말을 반복하고 있지 않은가?
- 새 섹션이 기존 섹션보다 더 빠르게 문제를 발견하게 만드는가?

증거:
- 대표 질문 3개에 대한 답변 가능 여부
- 중복 섹션 제거 검토 메모

### 3.4 Reliability Lens
검증 질문:
- 캐시가 오래된 데이터를 보여주지 않는가?
- 데이터 파일 일부가 없어도 전체 앱이 깨지지 않는가?
- 라이트 모드와 풀 모드가 서로 다른 진실을 보여주지 않는가?

증거:
- 캐시 무효화 로직 확인
- 일부 parquet 누락 상태 점검
- Lite/Full 동일 필터 비교

### 3.5 Maintainability Lens
검증 질문:
- 새 하드코딩이 생기지 않았는가?
- 상수/설정/렌더링/집계가 뒤섞이지 않았는가?
- 같은 계산이 여러 군데 복제되지 않았는가?

증거:
- `config/settings.example.yaml` 반영 여부
- 상수화 여부
- 중복 함수 탐색

### 3.6 Compliance Lens
검증 질문:
- 댓글 원문 노출 수준이 과도하지 않은가?
- 번역/요약 과정에서 민감 표현이 불필요하게 확장되지 않는가?
- 로그/다운로드에 개인정보 또는 식별자 위험이 없는가?

증거:
- 샘플 다운로드 검토
- 로그 문구 검토

## 4. Release Gates

대시보드 수정은 아래 게이트를 모두 통과해야 완료로 본다.

### Gate A. Functional Correctness
- [ ] 주요 필터 변경 시 KPI, 차트, 표가 함께 반응한다.
- [ ] 빈 결과에서 잘못된 이전 데이터가 남지 않는다.
- [ ] 다운로드 파일이 현재 필터 결과와 논리적으로 일치한다.

### Gate B. Insight Quality
- [ ] 첫 화면만 보고도 현재 문제 영역을 말할 수 있다.
- [ ] 대표 댓글과 키워드가 차트 해석을 보완한다.
- [ ] 중복 패널 없이 각 패널의 역할이 설명 가능하다.

### Gate C. Performance
- [ ] 첫 로드가 과도하게 느리지 않다.
- [ ] 필터 조작 시 브라우저가 멈춘 것처럼 느껴지지 않는다.
- [ ] Lite 모드는 실제로 빠르고, Full 모드는 실제로 더 깊다.

### Gate D. Harness Hygiene
- [ ] 새 설정값은 설정 파일 또는 상수에 반영했다.
- [ ] 변경 이유와 검증 방법을 문서 또는 PR 설명에 남겼다.
- [ ] 새 기술부채가 생기면 `docs/harness/exec-plans/tech-debt-tracker.md`에 기록했다.

## 5. Required Review Flow

대시보드 관련 작업은 항상 아래 순서로 진행한다.

1. 어떤 사용자 질문을 더 잘 답하려는 수정인지 적는다.
2. 영향받는 위젯과 집계 로직을 적는다.
3. 필터 일관성 관점에서 깨질 수 있는 지점을 적는다.
4. Lite/Full, 빈 상태, 다운로드를 최소 검증한다.
5. 남은 리스크와 기술부채를 문서에 남긴다.

## 6. Typical Failure Patterns

아래는 반복적으로 경계해야 하는 실패 패턴이다.

- 카드 숫자만 맞고 상세 표본은 다른 모집단을 보고 있음
- 표시용 한글 라벨과 내부 코드가 섞여 필터 누락 발생
- 빈 데이터인데 이전 캐시가 보여서 정상처럼 보임
- 대표 댓글이 너무 무거워서 전체 앱 반응성 저하
- 새 차트 추가로 정보량은 늘었지만 판단 속도는 느려짐
- Lite 모드가 빠르기만 하고 실제 검증 용도로는 쓸 수 없음
- 다운로드 파일이 화면 상태와 달라 검증 근거가 깨짐
